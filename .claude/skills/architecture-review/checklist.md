# 架构评估详细检查清单

> 本清单按评估优先级排列：**公司级规范合规（优先）→ 通用架构评估**

---

# 第一部分：公司级规范合规（优先）

## 组件合规性

### 状态定义

| 状态 | 含义 |
|------|------|
| **禁止** | 已被禁止使用在**任何**项目/产品中，必须移除并替换 |
| **退出** | 不在允许使用范围，现有项目应尽快移除 |
| **禁止新增** | 存量项目可暂时继续使用，但**不允许在新项目/产品中使用** |
| **允许例外** | 仅被允许在特定项目/产品中使用，需单独审批 |

### 检查项

**第一步：确认项目类型**（新项目 vs 存量项目，评判标准不同）
- [ ] 确认是新项目还是存量项目

**第二步：逐级核查**
- [ ] 【禁止】是否包含"禁止"状态的组件（如 fastjson、commons-lang、log4j 等）→ 任何项目必须移除
- [ ] 【退出】是否包含"退出"状态的组件 → 应尽快移除
- [ ] 【禁止新增】新项目是否引入了"禁止新增"的组件（如 hibernate、activiti 等）→ 新项目严禁使用
- [ ] 【禁止新增】存量项目的"禁止新增"组件是否有替换排期
- [ ] 【中间件-禁止】是否使用了禁止的中间件（如 codis、pika）→ 必须替换为 Redis
- [ ] 【中间件-禁止新增】是否使用了禁止新增的中间件（如 dubbo、eureka、tidb、memcached）
- [ ] 禁止/退出组件是否已有对应替换方案并排期整改

### 检查方法

```
1. 读取 data/*.json 禁止清单数据
2. 读取项目的 pom.xml / build.gradle / package.json 依赖声明
3. 交叉比对，找出违规项
4. 检查 docker-compose / yml 配置中的中间件引用
```

# 第二部分：通用架构评估

### 检查项

- [ ] 每个模块/包/命名空间是否有明确的单一职责
- [ ] 是否存在"上帝类"或"上帝模块"（承担过多职责）
- [ ] 业务逻辑、数据访问、展示层是否分离
- [ ] 工具类/公共模块是否职责单一且边界清晰
- [ ] 模块命名是否准确反映其职责

### 常见问题

| 问题 | 症状 | 建议 |
|------|------|------|
| 上帝模块 | 单文件超过 500 行，承担 3+ 职责 | 按职责拆分为多个模块 |
| 职责不清 | 模块名称模糊（如 utils、helper、common） | 按领域/功能重新组织 |
| 过度拆分 | 大量只有几行代码的模块，理解成本高 | 合并相关的细粒度模块 |

## 维度 2：依赖关系与耦合度

### 检查项

- [ ] 依赖方向是否单向（无循环依赖）
- [ ] 高层模块是否依赖抽象而非底层具体实现
- [ ] 模块间是否通过最小化接口通信
- [ ] 是否存在不必要的直接依赖（可通过事件/消息解耦）
- [ ] 第三方依赖是否被适当封装（防腐层）

### 耦合度等级（从低到高）

1. **无耦合**：模块间无直接关系
2. **数据耦合**：通过简单数据参数交互 ✅ 理想
3. **标记耦合**：通过复合数据结构交互 ⚠️ 可接受
4. **控制耦合**：传递控制标志影响对方行为 ⚠️ 需关注
5. **公共耦合**：共享全局数据 ❌ 应避免
6. **内容耦合**：直接访问对方内部数据 ❌ 必须消除

### 检查方法

```
1. 绘制模块依赖图，检查是否有环
2. 统计每个模块的传入/传出依赖数量
3. 检查 import/require 语句的方向性
4. 搜索全局变量和单例的使用
```

## 维度 3：分层架构合理性

### 检查项

- [ ] 是否有清晰的层次定义（文档或目录结构体现）
- [ ] 是否存在跨层调用（如 Controller 直接访问 Repository）
- [ ] 各层数据模型是否独立（不直接暴露数据库实体到 API）
- [ ] 层间是否有明确的 DTO/VO 转换
- [ ] 基础设施层是否与业务层解耦

### 常见分层模式

```
经典三层：
  Controller / Handler  →  Service  →  Repository / DAO

整洁架构：
  Frameworks & Drivers  →  Interface Adapters  →  Use Cases  →  Entities

六边形架构：
  Adapter (入站/出站)  →  Port (接口)  →  Domain (核心)
```

### 检查方法

```
1. 检查每层目录中的 import，确认只引用相邻层
2. 搜索 Repository/DAO 类在非 Service 层的使用
3. 检查数据库实体是否出现在 Controller 层
```

---

## 维度 4：扩展性与开放封闭

### 检查项

- [ ] 新增功能是否需要修改已有代码（违反 OCP）
- [ ] 是否有策略模式/工厂模式等扩展点
- [ ] 配置是否外化（不硬编码业务规则和阈值）
- [ ] 是否存在大量 if-else / switch-case 进行类型判断
- [ ] 是否过度设计（存在未使用的抽象层或扩展点）

### 常见问题

| 问题 | 症状 | 建议 |
|------|------|------|
| 类型膨胀 | 每种新类型都要修改 switch-case | 引入策略模式或多态 |
| 硬编码 | 业务阈值、URL、配置散落在代码中 | 提取到配置文件 |
| 过度设计 | 大量 Interface 只有一个实现 | 遵循 YAGNI，简化设计 |

---

## 维度 5：可测试性

### 检查项

- [ ] 核心业务逻辑是否可不启动完整应用即可测试
- [ ] 外部依赖（数据库、API、文件系统）是否可 Mock/Stub
- [ ] 是否使用依赖注入（构造函数注入优先）
- [ ] 函数是否无副作用或副作用被隔离
- [ ] 测试金字塔比例是否合理（大量单元 > 适量集成 > 少量 E2E）

### 检查方法

```
1. 检查构造函数参数，是否接受接口而非具体类
2. 搜索 new 关键字在业务逻辑中的使用
3. 检查静态方法/全局函数的使用频率
4. 查看现有测试的类型分布
```

---

## 维度 6：复杂度管理

### 检查项

- [ ] 函数行数是否控制在 30 行以内（建议值）
- [ ] 类的行数是否控制在 300 行以内（建议值）
- [ ] 嵌套深度是否不超过 3 层
- [ ] 同一函数内是否混合了不同抽象层级的操作
- [ ] 是否有超长的参数列表（5+ 参数）

### 复杂度阈值参考

| 指标 | 良好 | 警告 | 危险 |
|------|------|------|------|
| 函数行数 | < 30 | 30-60 | > 60 |
| 类行数 | < 300 | 300-500 | > 500 |
| 圈复杂度 | < 10 | 10-20 | > 20 |
| 嵌套深度 | ≤ 2 | 3 | ≥ 4 |
| 参数个数 | ≤ 3 | 4-5 | ≥ 6 |

---

## 维度 7：错误处理与弹性

### 检查项

- [ ] 错误处理策略是否全局一致（异常 / 错误码 / Result 类型）
- [ ] 是否存在空的 catch 块（静默吞异常）
- [ ] 外部调用是否有超时设置
- [ ] 是否有重试机制（带退避策略）
- [ ] 是否有降级/兜底方案
- [ ] 资源是否在 finally/defer 中正确释放

### 检查方法

```
1. 搜索空的 catch/except 块
2. 搜索外部 HTTP/RPC 调用，检查超时配置
3. 检查数据库连接/文件句柄是否正确关闭
4. 搜索 TODO/FIXME 中关于错误处理的标记
```

---

## 维度 8：API 与接口设计

### 检查项

- [ ] 公开 API 是否有明确的版本管理策略
- [ ] 接口参数是否最小化（不传递不需要的数据）
- [ ] 返回值是否自解释（不依赖调用者了解内部状态）
- [ ] 是否遵循接口隔离原则（不强迫依赖方实现不需要的方法）
- [ ] 内部模块间接口是否稳定

---

## 维度 9：数据流与状态管理

### 检查项

- [ ] 数据流向是否单向且可追踪
- [ ] 全局状态是否最小化
- [ ] 共享可变状态是否有同步保护
- [ ] 副作用是否被隔离到边界层
- [ ] 配置/环境变量的读取是否集中管理

### 检查方法

```
1. 搜索全局变量/单例的定义和使用
2. 检查并发场景下的共享资源访问
3. 追踪关键数据从输入到输出的完整路径
```

---

## 维度 10：安全架构

### 检查项

- [ ] 认证（Authentication）是否在统一入口处理
- [ ] 鉴权（Authorization）是否细粒度且集中管理
- [ ] 用户输入是否在入口层进行校验和清洗
- [ ] 敏感数据是否加密存储和传输
- [ ] 日志中是否脱敏处理敏感信息
- [ ] 是否遵循最小权限原则

---

## 维度 11：可观测性

### 检查项

- [ ] 是否有结构化日志（非随意 print/console.log）
- [ ] 关键业务流程是否有完整的日志覆盖
- [ ] 是否有请求链路追踪（traceId）
- [ ] 是否有业务和系统指标的采集
- [ ] 是否有健康检查端点
- [ ] 告警规则是否合理

---

## 维度 12：性能架构

### 检查项

- [ ] 热点路径是否有缓存策略
- [ ] 数据库查询是否有 N+1 问题
- [ ] 连接池/线程池是否合理配置
- [ ] 大数据量场景是否有分页/流式处理
- [ ] 是否有异步处理机制（耗时操作不阻塞主流程）
- [ ] 资源是否及时释放（连接、句柄、临时文件）
