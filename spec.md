## 0. 核心用例

### 0.1 运营人员配置定价用例

#### UC-0.1.1 配置 Token 按量计费定价

**用例名称**：配置 Token 按量计费定价规则

**参与者**：运营人员

**前置条件**：
- 运营人员已登录系统
- 运营人员具有定价配置权限
- 系统中已存在可用的模型列表

**基本流程**：
1. 运营人员进入定价配置页面
2. 选择资源类型为 Token，计费模式为按量计费
3. 选择要配置的模型（如 GPT-4）
4. 设置输入 Token 单价（如 0.01 元/千 token）
5. 设置输出 Token 单价（如 0.03 元/千 token）
6. 配置阶梯计费规则（可选）
   - 第一阶梯：0-100 万 token，单价 x
   - 第二阶梯：100-500 万 token，单价 y
   - 第三阶梯：500 万以上，单价 z
7. 设置定价有效期（如：长期有效或指定时间范围）
8. 点击保存
9. 系统验证定价配置的合法性
10. 系统保存定价配置并返回成功提示

**扩展流程**：
- **4a. 单价格式错误**：系统提示单价格式不正确，要求重新输入
- **9a. 配置冲突**：系统提示与现有配置冲突，询问是否覆盖
- **9b. 阶梯配置错误**：系统提示阶梯范围不连续或重叠，要求重新配置

**后置条件**：
- Token 定价配置已保存到系统
- 新的定价立即对后续消费生效
- 定价配置变更被记录到审计日志

### 0.2 计费计算用例

#### UC-0.2.1 智能体调用 Token 计费计算

**用例名称**：智能体调用时 Token 消费计费计算

**参与者**：系统

**前置条件**：
- 用户已登录
- 用户账户余额充足
- Token 定价配置已生效

**基本流程**：
1. 用户发起智能体调用请求
2. 系统预估本次调用所需的 Token 数量（输入 token）
3. 系统根据定价配置预估费用
4. 系统检查用户余额是否充足
5. 系统从用户账户预扣预估费用
6. 智能体执行，产生实际的输入和输出 token
7. 系统计算实际费用：
   - 获取对应模型的定价配置
   - 应用阶梯计费规则（如适用）
   - 计算输入 token 费用
   - 计算输出 token 费用
   - 总费用 = 输入费用 + 输出费用
8. 系统对比预扣费用与实际费用
9. 若预扣费用 > 实际费用，退还差额到用户账户
10. 若预扣费用 < 实际费用，从用户账户补扣差额
11. 系统记录消费明细（用户、租户、资源类型、数量、费用、时间）
12. 系统检查是否触发消费预警
13. 返回调用结果给用户

**扩展流程**：
- **4a. 余额不足**：系统拒绝服务，返回余额不足提示
- **10a. 补扣余额不足**：系统记录差额，不扣减余额，但记录实际消费
- **12a. 触发预警**：系统发送邮件预警给用户和管理员
- **12b. 超出预算**：系统停止用户服务，返回超限提示

**后置条件**：
- Token 消费已正确计费
- 费用归属到用户和租户
- 消费明细已记录
- 余额已正确更新

#### UC-0.2.2 阶梯计费计算示例

**用例名称**：阶梯计费计算逻辑

**场景**：Token 按量计费，配置三阶梯规则

**定价配置**：
- 第一阶梯：0-100 万 token，0.01 元/千 token
- 第二阶梯：100-500 万 token，0.008 元/千 token
- 第三阶梯：500 万以上，0.005 元/千 token

**计算示例**：

**示例1**：使用 50 万 token
- 全部落在第一阶梯
- 费用 = 500,000 / 1,000 × 0.01 = 5 元

**示例2**：使用 200 万 token
- 前 100 万在第一阶梯：1,000,000 / 1,000 × 0.01 = 10 元
- 后 100 万在第二阶梯：1,000,000 / 1,000 × 0.008 = 8 元
- 总费用 = 10 + 8 = 18 元

**示例3**：使用 600 万 token
- 前 100 万在第一阶梯：10 元
- 100-500 万（400 万）在第二阶梯：4,000,000 / 1,000 × 0.008 = 32 元
- 500 万以上（100 万）在第三阶梯：1,000,000 / 1,000 × 0.005 = 5 元
- 总费用 = 10 + 32 + 5 = 47 元

#### UC-0.2.3 OSS 叠加计费计算

**用例名称**：OSS 流量和请求次数叠加计费

**场景**：用户在一个月内使用了 OSS

**定价配置**：
- 流量计费：0.1 元/GB
- 请求计费：0.01 元/千次

**使用情况**：
- 上传流量：10 GB
- 下载流量：20 GB
- 读请求：5,000 次
- 写请求：2,000 次

**计算过程**：
1. 流量费用 = (10 + 20) × 0.1 = 3 元
2. 请求费用 = (5,000 + 2,000) / 1,000 × 0.01 = 0.07 元
3. 总费用 = 3 + 0.07 = 3.07 元

**后置条件**：
- 流量费用和请求费用分别记录
- 总费用正确累加

#### UC-0.2.4 包月与按量混合计费

**用例名称**：同一用户不同资源类型采用不同计费模式

**场景**：用户 A 采用以下计费配置
- Token：包月计费，99 元/月，额度 100 万 token
- OSS：按量计费
- 外部 API：按量计费

**使用使用情况**：
- Token 实际使用：80 万 token
- OSS 流量：5 GB
- 外部 API 调用：1,000 次

**计费计算**：
1. Token 费用：包月计费，固定 99 元（未超额度）
2. OSS 费用：5 × 0.1 = 0.5 元
3. 外部 API 费用：1,000 / 1,000 × 0.05 = 0.05 元
4. 总费用 = 99 + 0.5 + 0.05 = 99.55 元

#### UC-0.4.4 并发余额扣减冲突

**用例名称**：并发请求时余额扣减的冲突处理

**参与者**：系统

**前置条件**：
- 用户账户余额 10 元
- 用户同时发起多个智能体调用

**场景**：
- 请求1：预估费用 6 元
- 请求2：预估费用 5 元
- 请求3：预估费用 4 元

**基本流程**：
1. 请求1-3 同时到达
2. 系统对用户余额加锁
3. 请求1 获取锁，检查余额 10 元，预扣 6 元，余额变为 4 元，释放锁
4. 请求2 获取锁，检查余额 4 元，预估费用 5 元 > 余额 4 元，拒绝服务，释放锁
5. 请求3 获取锁，检查余额 4 元，预估费用 4 元 ≤ 余额 4 元，预扣 4 元，余额变为 0 元，释放锁
6. 系统返回各请求结果：
   - 请求1：成功
   - 请求2：失败，余额不足
   - 请求3：成功

**扩展流程**：
- 无

**后置条件**：
- 余额正确：0 元
- 请求1和请求3 成功执行
- 请求2 被拒绝
- 无数据不一致

9. 账单生成成功
10. 系统更新任务状态为成功

**扩展流程**：

1. **用户发起调用**

   - 系统预估本次调用所需资源
   - 系统根据定价规则预估费用
   - 系统检查用户余额
   - 系统预扣预估费用

   - 智能体调用外部 API，产生 API 调用费用

   - 系统向用户返回费用信息

- **6a. 触发预警**：系统发送预警邮件
- **6b. 超出预算**：系统停止用户服务

**后置条件**：
- 智能体调用完成
- 费用正确计费
- 消费明细已记录
- 用户和租户消费已更新

## 1. 用户故事

- 不同资源类型可以独立设置计费模式
- 计费周期统一为自然月（每月1日至月底）

**US-1.1.2 作为普通用户，我希望实时查看自己的消费情况**

**描述**：用户可以随时查看当月各资源类型的累计消费金额和数量。

**验收标准**：

- 用户可以查看自己所属的租户信息

### 1.2 Token计费

**US-1.2.1 作为系统，需要支持按Token输入和输出分别计费**

**描述**：系统需要区分智能体使用的输入Token和输出Token，并分别计算费用。

**验收标准**：
- 输入Token和输出Token可以设置不同的单价
- 不同模型可以使用不同的Token单价
- 运营人员可以为不同模型配置Token定价
- Token消耗量按实际使用量累计
- 支持查看Token消耗的详细记录（模型、输入量、输出量、费用）

**US-1.2.2 作为运营人员，可以为Token配置包月或按量计费模式**

**描述**：运营人员可以为 Token 资源类型配置包月计费或按使用量计费模式。

**验收标准**：
- 包月计费：支付固定月费，Token使用不限量或按套餐额度使用
- 按量计费：根据实际Token使用量乘以单价计算费用
- 可以针对输入Token和输出Token分别配置计费模式
- 可以针对不同模型分别配置计费模式
- 同一资源类型支持配置多个定价策略

**US-1.2.3 作为运营人员，可以配置Token阶梯计费规则**

**描述**：运营人员可以为 Token 配置多个计费阶梯，使用量达到不同阶梯时应用不同单价。

**验收标准**：
- 支持配置多个计费阶梯（如：前100万Token一个价，100万-500万一个价，超过500万一个价）
- 阶梯可以针对输入Token和输出Token分别设置
- 阶梯可以针对不同模型分别设置
- 系统自动根据累计使用量选择对应的阶梯单价
- 阶梯计费与包月计费互斥

### 1.3 OSS存储计费

**US-1.3.1 作为系统，需要支持OSS按流量计费**

**描述**：系统根据智能体使用OSS的上传和下载流量计算费用。

**验收标准**：
- 支持配置上传流量和下载流量的单价
- 支持总流量统一计费或上下行分别计费
- 流量按实际使用量累计（GB/MB）
- 运营人员可以配置OSS流量计费规则
- 支持查看流量消耗的详细记录

**US-1.3.2 作为系统，需要支持OSS按请求次数计费**

**描述**：系统根据智能体对OSS的API请求次数计算费用。

**验收标准**：
- 支持配置每次请求的单价
- 可以区分读请求和写请求设置不同单价
- 请求次数按实际调用次数累计
- 运营人员可以配置OSS请求次数计费规则
- 支持查看请求次数的详细记录

**US-1.3.3 作为系统，需要支持OSS叠加计费**

**描述**：OSS可以同时按流量和请求次数计费，最终费用为两者之和。

**验收标准**：
- 支持同时启用流量计费和请求次数计费
- 最终费用 = 流量费用 + 请求费用
- 两种计费方式可以独立启用或禁用
- 账单中分别显示两种方式产生的费用

**US-1.3.4 作为运营人员，可以为OSS配置包月或按量计费模式**

**描述**：运营人员可以为 OSS 资源类型配置包月计费或按使用量计费模式。

**验收标准**：
- 包月计费：支付固定月费，OSS使用不限量或按套餐额度使用
- 按量计费：根据实际流量和请求次数计算费用
- 可以针对流量计费和请求次数计费分别配置计费模式
- 包月计费可以针对流量、请求次数或两者都设置

### 1.4 外部API调用计费

**US-1.4.2 作为运营人员，可以为外部API配置包月或按量计费模式**

**描述**：运营人员可以为外部 API 资源类型配置包月计费或按使用量计费模式。

**验收标准**：

## 3. 边界情况

### 3.1 计费相关边界情况

**BC-3.1.1 零使用量计费**
- 用户在某月未使用任何智能体，不产生费用
- 某资源类型未使用，该资源费用为0
- 包月计费即使未使用也收取固定费用
- 零使用量不影响账单生成

**BC-3.1.2 超大使用量计费**
- 单次调用使用Token数量极大时正确计费
- 单月累计Token数量极大时正确计费
- 计费数据不溢出
- 费用计算正确

**BC-3.1.3 阶梯边界**
- 使用量正好在阶梯边界时应用正确单价
- 跨阶梯使用量正确分段计算
- 多个阶梯规则优先级明确
- 阶梯配置错误时有默认处理

**BC-3.1.4 多资源叠加**
- 同一次智能体调用使用多种资源，费用正确累加
- 不同资源计费失败不影响其他资源计费
- 总费用等于各资源费用之和
- 账单中各资源费用独立显示
- 各资源费用归属到用户和租户

**BC-3.1.5 模型变更**
- 智能体运行过程中切换模型，不同模型Token分别计费
- 历史记录中模型信息准确
- 已停用模型的计费记录仍可查询
- 新模型默认使用运营人员配置的默认单价
- 模未配置定价时使用系统默认定价

### 3.2 预付费相关边界情况

**BC-3.2.1 余额不足**
- 余额为0时拒绝服务
- 余额不足预估费用时拒绝服务
- 多次并发请求余额不足时全部拒绝
- 提示信息明确说明原因

**BC-3.2.2 预扣与实际差异**
- 实际费用小于预扣费用，差值退回余额
- 实际费用大于预扣费用，从余额补扣
- 余额不足以补扣时处理机制明确
- 退款到账立即生效

**BC-3.2.3 充值时机**
- 服务过程中充值，余额及时更新
- 充值金额可以复用
- 充值失败不影响已扣费用
- 充值记录准确完整
- 充值金额归属到用户和租户

**BC-3.2.4 负余额处理**
- 不允许出现负余额
- 补扣余额不足时不扣减余额但有记录
- 账单显示实际消费金额
- 差异金额有明确标识

### 3.3 优惠相关边界情况

**BC-3.3.1 优惠重叠**
- 多个优惠同时满足条件时应用最优优惠
- 满减优惠和折扣优惠优先级明确
- 优惠叠加规则明确
- 不允许重复应用同一优惠

**BC-3.3.2 优惠时效**
- 优惠活动开始前不生效
- 优惠活动结束后立即失效
- 跨月的优惠活动正确处理
- 限时优惠到期自动失效

**BC-3.3.3 优惠与预算冲突**
- 优惠减免后的费用仍可能触发预算预警
- 优惠金额不计入预算消耗
- 预算上限基于优惠后费用判断
- 提示信息中包含优惠信息

**BC-3.3.4 优惠取消**
- 优惠活动取消后已应用的优惠不追溯
- 新消费不再应用已取消的优惠
- 优惠记录保留历史信息
- 不影响账单准确性

**BC-3.3.5 优惠与租户**
- 优惠可以针对全部租户生效
- 优惠可以针对指定租户生效
- 未指定租户时默认对全部租户生效
- 租户删除后优惠配置保留

### 3.4 账单相关边界情况

**BC-3.4.1 首次账单**
- 新用户首月账单正确生成
- 新租户首月账单正确生成
- 无消费时账单金额为0
- 账单数据完整
- 账单归属关系正确

**BC-3.4.2 跨月处理**
- 跨月消费正确归属到对应月份
- 月底最后时刻消费正确记录
- 月初第一时刻消费正确记录
- 账单周期边界清晰
- 费用归属到正确的月份

**BC-3.4.3 历史数据缺失**
- 历史数据缺失时账单生成不中断
- 缺失数据有明确标识
- 可手动补充数据后重新生成账单
- 不影响正常计费

**BC-3.4.4 账单修改**
- 已出账单不支持直接修改
- 需要调整时通过红字账单处理
- 调整记录完整保存
- 不影响历史账单准确性

**BC-3.4.5 账单归属**
- 账单按租户维度汇总
- 明细按用户维度展示
- 每笔费用明确归属到用户和租户
- 租户下用户迁移时账单归属正确

### 3.5 发票相关边界情况

**BC-3.5.1 发票重开**
- 已开发票的账单不支持重复开票
- 需要重开时先作废旧发票
- 作废发票有明确记录
- 重开发票序号连续

**BC-3.5.2 发票金额调整**
- 账单金额调整后发票金额不一致
- 需要重开发票
- 差异金额有明确说明
- 不影响税收合规

**BC-3.5.3 发票信息错误**
- 发票信息错误时可以申请作废
- 作废后可重新开具
- 错误信息有记录
- 不影响合规性

**BC-3.5.4 发票归属**
- 发票归属到租户
- 租户管理员可以申请发票
- 运营人员可以为租户代开发票
- 发票与账单的租户归属一致

### 3.6 系统异常边界情况

**BC-3.6.1 计费服务中断**
- 计费服务中断时智能体调用失败
- 恢复后自动恢复服务
- 中断期间消费数据不丢失
- 可手动补充计费数据

**BC-3.6.2 数据库异常**

- 并发生成账单不重复

**BC-3.6.4 网络异常**

- 网络异常时计费记录本地缓存
- 网络恢复后自动上传缓存数据
- 不丢失计费数据

### 3.7 配置相关边界情况

**BC-3.7.1 定价配置缺失**

- 新资源类型缺少定价时使用默认配置
